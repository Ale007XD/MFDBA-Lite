1. Архитектура системы

1.1 FSM-детерминизм

Система должна быть детерминированной и описываться функцией перехода:

[
\delta(S, E) \rightarrow S'
]

где

- S — текущее состояние
- E — событие
- S' — следующее состояние

Запрещено:

- неявные циклы
- глобальные изменяемые переменные
- логика, нарушающая воспроизводимость состояний.

---

1.2 Stateless Engine

"ExecutionEngine.step()" обязан быть идемпотентным.

Все промежуточные данные и контекст выполнения хранятся исключительно в:

StateContext

Engine не должен хранить скрытое состояние.

---

1.3 Chain of Responsibility

Роутинг и обработка запросов строятся через цепочки middleware.

Запрещено:

if / else роутинг
switch-based dispatch
глубокие ветвления логики

Используется pipeline-подход:

Request → Middleware → Middleware → Handler

---

1.4 BaseProvider (ABC)

Любой LLM-провайдер обязан наследоваться от:

mfdballm.providers.base.BaseProvider

И реализовывать канонический метод:

async def chat(
    self,
    messages: List[dict],
    tools: list | None = None
) -> ProviderResponse

---

1.5 Нормализация данных

Роутер обязан возвращать нормализованный объект:

ProviderResponse

Запрещено внутри движка:

- "str"
- "dict"
- любые «сырые» ответы провайдеров.

---

1.6 Строгая типизация

Используется Pydantic v2.

Все входящие и исходящие данные проходят через:

pydantic.BaseModel

с обязательной валидацией схем.

---

1.7 Minimalism

Ключевые компоненты ядра (например Router) должны оставаться компактными.

Целевой размер:

≈ 30–80 строк

Избыточные уровни абстракции запрещены.

---

1.8 Async / Sync совместимость

Система должна поддерживать оба режима выполнения.

Определение контекста выполняется через:

asyncio.get_running_loop()

---

2. Процесс разработки

2.1 Interface-First

Порядок разработки:

1. определить контракт
2. реализовать ABC / Pydantic модели
3. написать реализацию

Реализация не может предшествовать контракту.

---

2.2 Интерфейсы превыше реализации

Если возникает конфликт:

старый тест vs новый контракт

допустимы только два решения:

- исправить тест
- добавить adapter / shim

Контракт менять запрещено.

---

2.3 Политика тестов

Тесты должны соответствовать актуальной архитектуре.

Если тест остаётся зелёным только благодаря костылям:

→ тест рефакторится

а не поддерживается.

---

2.4 Консистентность JSON

Каждая техническая задача должна сопровождаться:

session_summary
decisions_log

в валидном JSON формате.

---

3. Правила технической коммуникации

3.1 Технический стиль

Стиль общения:

«производственный тех-коллаборатор»

Требования:

- без воды
- без поучений
- коротко и структурировано.

---

3.2 Математическая строгость

Описание логики и состояний должно использовать LaTeX.

Пример:

[
S_{t+1} = \delta(S_t, E_t)
]

---

3.3 Структура ответов

Обязательно использовать:

- заголовки
- списки
- таблицы (при необходимости)

Запрещены «стены текста».

---

3.4 Следующий шаг

Каждый ответ должен завершаться конкретным действием:

Would you like me to ...

---

4. Архитектурная цель

Система MFDB должна обладать следующими свойствами:

Свойство| Гарантия
Детерминизм| FSM-архитектура
Воспроизводимость| идемпотентный Engine
Типобезопасность| Pydantic v2
Провайдер-агностичность| BaseProvider
Предсказуемость| ProviderResponse
Простота| Minimalism
